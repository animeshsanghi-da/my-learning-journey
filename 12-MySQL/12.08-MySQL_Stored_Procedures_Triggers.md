# MySQL Database Programming: Stored Procedures & Triggers

## Overview
Database programming allows you to move logic from the application layer directly into the database. This improves performance, ensures data consistency, and allows for the reuse of complex SQL logic. As covered in the **Ultimate MySQL Handbook**, Stored Procedures and Triggers are the two primary tools for this automation.

---

## 1. Stored Procedures
A stored procedure is a prepared SQL code that you can save, so the code can be reused over and over again. It encapsulates logic that can include queries, updates, and conditional operations.

### Why use Stored Procedures?
* **Encapsulation:** Group complex logic into a single callable unit.
* **Reduced Network Traffic:** Multiple commands are executed in one call to the server.
* **Security:** You can give users permission to execute a procedure without giving them direct access to the underlying tables.

### The Delimiter Concept
By default, MySQL uses `;` to end a statement. Since a procedure contains multiple statements ending in `;`, we must temporarily change the delimiter (e.g., to `$$` or `//`) so MySQL treats the entire block as one unit.

### Basic Syntax & Example
To create a procedure that adds a new user to the `users` table:

```sql
DELIMITER $$

CREATE PROCEDURE AddUser(
    IN p_name VARCHAR(100),
    IN p_email VARCHAR(100),
    IN p_gender ENUM('Male', 'Female', 'Other'),
    IN p_dob DATE,
    IN p_salary INT
)
BEGIN
    INSERT INTO users (name, email, gender, date_of_birth, salary)
    VALUES (p_name, p_email, p_gender, p_dob, p_salary);
END$$

DELIMITER ;
```

### Managing Procedures
* **Calling a Procedure:**
  ```sql
  CALL AddUser('Kiran Sharma', 'kiran@example.com', 'Female', '1994-06-15', 72000);
  ```
* **Viewing Procedures:**
  ```sql
  SHOW PROCEDURE STATUS WHERE Db = 'startersql';
  ```
* **Deleting a Procedure:**
  ```sql
  DROP PROCEDURE IF EXISTS AddUser;
  ```

## 2. Triggers
A trigger is a named database object that is associated with a table and activates when a particular event (INSERT, UPDATE, or DELETE) occurs for the table.

### Use Cases for Triggers
* **Auditing/Logging:** Automatically record who changed what and when.
* **Data Validation:** Enforce complex rules that simple constraints cannot handle.
* **Derived Data:** Automatically update a "Total Sales" column in a summary table when a new order is placed.

### Trigger Components
* **Timing:** `BEFORE` or `AFTER` the event.
* **Event:** `INSERT`, `UPDATE`, or `DELETE`.
* **NEW vs OLD:** * `NEW.column`: Refers to the data being inserted or the new version after an update.
    * `OLD.column`: Refers to the data before an update or the data being deleted.

### Trigger Example: Automated Logging
Suppose we want to log every time a new user is created. First, we create a log table:

```sql
CREATE TABLE user_log (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT,
    name VARCHAR(100),
    created_on TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

Then, create the trigger:

```sql
DELIMITER $$

CREATE TRIGGER after_user_insert
AFTER INSERT ON users
FOR EACH ROW
BEGIN
    INSERT INTO user_log (user_id, name)
    VALUES (NEW.id, NEW.name);
END$$

DELIMITER ;
```

### Managing Triggers
* **Deleting a Trigger:**
  ```sql
  DROP TRIGGER IF EXISTS after_user_insert;
  ```

## 3. Summary of Comparison

| Feature | Stored Procedure | Trigger |
| :--- | :--- | :--- |
| **Execution** | Called explicitly using `CALL` | Fired automatically by events |
| **Parameters** | Accepts `IN`, `OUT`, and `INOUT` | Does not accept parameters |
| **Logic Type** | Business logic, complex workflows | Data integrity, auditing, synchronization |
| **Control** | User-controlled | System-controlled |

---

## 4. Best Practices
1. **Keep Triggers Simple:** Since triggers run automatically on every row affected, complex logic inside a trigger can significantly slow down your database.
2. **Naming Conventions:** Use clear names like `idx_` for indexes, `trg_` for triggers, and `sp_` for stored procedures to keep the schema organized.
3. **Transaction Safety:** Stored procedures can be used to wrap multiple statements in a transaction (using `COMMIT` and `ROLLBACK`) to ensure data integrity.
