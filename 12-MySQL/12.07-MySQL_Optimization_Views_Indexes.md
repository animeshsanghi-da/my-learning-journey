# MySQL Optimization: Views, Indexes, and Transactions

## Overview
As data grows from thousands to millions of rows, query performance becomes a critical factor. This module covers how to simplify complex data access using **Views**, speed up data retrieval using **Indexes**, and ensure data integrity using **Transactions**.

---

## 1. MySQL Views
A **View** is a virtual table based on the result-set of an SQL statement. It does not store data itself but acts as a "live snapshot" of the underlying base tables.

### Why Use Views?
* **Simplify Queries:** Encapsulate complex joins and calculations into a single virtual table.
* **Security:** Provide users access to specific columns while hiding sensitive data (e.g., hiding 'salary' but showing 'name').
* **Consistency:** Ensure the same logic is reused across different parts of an application.

### Practical Implementation
```sql
-- Creating a view for high-earning users
CREATE VIEW high_salary_users AS
SELECT id, name, salary
FROM users
WHERE salary > 70000;

-- Querying the view just like a real table
SELECT * FROM high_salary_users;

-- Updating base data reflects in the view automatically
UPDATE users SET salary = 72000 WHERE name = 'Raj';
SELECT * FROM high_salary_users; -- 'Raj' now appears in the results

-- Removing a view
DROP VIEW high_salary_users;
```

## 2. MySQL Indexes
Indexes are used to find rows with specific column values quickly. Without an index, MySQL must begin with the first row and then read through the entire table to find the relevant rows (Full Table Scan).

### Types of Indexes
1.  **Primary Key Index:** Automatically created for the `PRIMARY KEY`. It is unique and does not allow NULLs.
2.  **Single-Column Index:** Created on a single column frequently used in `WHERE` or `JOIN` clauses.
3.  **Multi-Column Index:** A combined index on multiple columns, useful when queries filter by more than one field simultaneously.

### Managing Indexes
```sql
-- View all indexes on a table
SHOW INDEXES FROM users;

-- Create a single-column index on 'email' to speed up searches
CREATE INDEX idx_email ON users (email);

-- Create a multi-column index on 'gender' and 'salary'
CREATE INDEX idx_gender_salary ON users (gender, salary);

-- Drop an index
DROP INDEX idx_email ON users;
```

### Index Best Practices
* **Use Indexes on:** Columns used in `WHERE` clauses, `JOIN` conditions, and `ORDER BY` statements.
* **Avoid Indexes on:** Columns with very few unique values (like 'Gender') or tables that are frequently updated (as indexing slows down `INSERT`, `UPDATE`, and `DELETE` operations).

## 3. MySQL Transactions and AutoCommit
A **Transaction** is a sequential group of database operations performed as a single logical unit of work.

### ACID Properties
* **Atomicity:** All operations succeed, or none do.
* **Consistency:** The database remains in a valid state.
* **Isolation:** Transactions do not interfere with each other.
* **Durability:** Once committed, changes are permanent.

### Transaction Control
By default, MySQL uses `autocommit = 1`, meaning every statement is saved immediately. To perform multiple operations safely, you can manage them manually.

```sql
-- Disable AutoCommit
SET autocommit = 0;

-- Example Workflow: Updating a user's salary
UPDATE users SET salary = 80000 WHERE id = 5;

-- If satisfied, save changes permanently
COMMIT;

-- If an error occurred, revert changes to the last safe point
ROLLBACK;

-- Re-enable AutoCommit
SET autocommit = 1;
```
