# Virtual Environments & Dependency Management

## Overview
When working on Python and Data Science projects, you will often rely on external libraries (like Pandas, NumPy, or Scikit-learn). As you build multiple projects, you will encounter situations where Project A requires version 1.0 of a library, but Project B requires version 2.0. 

**Virtual Environments** solve this problem by creating isolated, self-contained workspaces for each project. **Dependency Management** ensures that anyone else (or you on a different computer) can recreate the exact environment needed to run your code without version conflicts.

---

## 1. What is a Virtual Environment?
A virtual environment is a directory tree that contains a localized Python installation, plus a number of additional packages.
* **Isolation:** Packages installed in one virtual environment do not affect the global Python installation or other virtual environments.
* **Reproducibility:** You can easily document exactly which packages and versions your project needs.
* **Cleanliness:** Keeps your global operating system environment clean and stable.

---

## 2. Managing Environments with `venv` (Standard Python)
The `venv` module is built directly into Python 3 and is standard for general Python development.

### Creating a Virtual Environment
Navigate to your project folder in your terminal and run:
```bash
# Syntax: python -m venv <environment_name>
python -m venv myenv
```

### Activating the Environment
Before installing packages or running your code, you must activate the environment.
```bash
# On Windows:
myenv\Scripts\activate

# On macOS and Linux:
source myenv/bin/activate
```
*(Once activated, your terminal prompt will change to show `(myenv)` at the beginning).*

### Deactivating the Environment
When you are done working on the project, simply type:
```bash
deactivate
```

---

## 3. Managing Environments with Conda (Data Science Standard)
In Data Science, **Anaconda** and **Miniconda** are highly preferred because they manage both Python packages and external C-libraries seamlessly. 

### Creating a Conda Environment
```bash
# Create an environment with a specific Python version
conda create --name ds_env python=3.10
```

### Activating and Deactivating Conda Environments
```bash
# Activate the environment
conda activate ds_env

# Deactivate the environment
conda deactivate
```

### Listing Environments
To see all Conda environments on your machine:
```bash
conda env list
```

---

## 4. Dependency Management: Tracking Your Packages
Once your environment is active, you need to install the libraries your project requires and track them so others can replicate your work.

### Using `pip` and `requirements.txt`
`pip` is the standard package installer for Python.
```bash
# Installing packages
pip install pandas numpy matplotlib

# View installed packages
pip list

# Exporting dependencies to a text file
pip freeze > requirements.txt
```

**Recreating the environment:** If someone downloads your project from GitHub, they can install all the required dependencies in one go:
```bash
pip install -r requirements.txt
```

### Using Conda and `environment.yml`
If you are using Conda, it is best practice to export an `environment.yml` file, which tracks both `conda` and `pip` installations.
```bash
# Exporting the Conda environment
conda env export > environment.yml

# Recreating the Conda environment from the YAML file
conda env create -f environment.yml
```

---

## 5. Best Practices & GitHub Integration
1. **Never commit environments to GitHub:** Virtual environment folders (like `myenv/` or `venv/`) contain thousands of files and are OS-specific. Always add your environment folder name to your `.gitignore` file.
2. **Always commit `requirements.txt` or `environment.yml`:** This is the blueprint of your project. It is essential for version control and collaboration.
3. **One environment per project:** Do not use a single giant environment for all your projects. Create a fresh, lightweight environment for each new repository.
