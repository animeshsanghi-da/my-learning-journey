# Git Branching and Merging: Collaborative Development

## Overview
Branching is one of Git's most powerful features. it allows you to diverge from the main line of development and continue to work without affecting that main line. In a Data Science context, this is essential for testing new models, cleaning data in a sandbox, or developing features without breaking the "production" code.

---

## 1. Understanding Branches
A branch in Git is simply a lightweight moveable pointer to one of your commits. The default branch name in Git is `main` (or `master`).

* **Main Branch:** The definitive, stable version of your project.
* **Feature Branches:** Temporary branches created to work on a specific task (e.g., `data-cleaning`, `linear-regression-model`).

---

## 2. Working with Branches
Common commands to manage your development timeline:

### Creating and Switching
```bash
# Create a new branch
git branch feature-name

# Switch to the new branch
git checkout feature-name

# Shortcut: Create and switch in one command
git checkout -b new-analysis
```

### Listing and Deleting
```bash
# List all local branches (* indicates the current branch)
git branch

# Delete a branch (after it has been merged)
git branch -d feature-name

# Force delete a branch (if not merged)
git branch -D old-experiment
```

---

## 3. Merging Branches
Once your work on a feature branch is complete and tested, you need to integrate it back into the `main` branch.

### The Merge Workflow
1.  Switch to the receiving branch (usually `main`).
2.  Run the merge command.

```bash
git checkout main
git merge feature-name
```

### Types of Merges
* **Fast-Forward Merge:** Occurs if the `main` branch hasn't moved since you created the feature branch. Git simply moves the pointer forward.
* **Three-Way Merge:** Occurs if `main` has new commits. Git creates a new "merge commit" that ties the two histories together.

---

## 4. Handling Merge Conflicts
A conflict occurs when changes are made to the same line of the same file in two different branches, and Git doesn't know which version to keep.

### Resolving a Conflict
1.  Git will pause the merge and mark the files as "unmerged."
2.  Open the affected file. You will see markers:
    ```text
    <<<<<<< HEAD
    StandardScaler()  # Change in main
    =======
    MinMaxScaler()    # Change in feature branch
    >>>>>>> feature-name
    ```
3.  Manually edit the file to keep the desired code and remove the markers.
4.  Add and commit the resolved file:
    ```bash
    git add .
    git commit -m "Fixed merge conflict in preprocessing script"
    ```

---

## 5. Branching Best Practices for Data Science
* **Keep Main Clean:** Only merge code that is tested and functional.
* **Descriptive Names:** Use names like `fix-null-values` or `add-visualization` instead of `test1`.
* **Merge Often:** Regularly merge `main` into your feature branch to stay updated and reduce the complexity of the final merge.
