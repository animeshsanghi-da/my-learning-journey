# Power BI: Data Extraction & Transformation using Power Query

## Overview
Before data can be visualized in Power BI, it must be extracted from its source, cleaned, and shaped into a usable format. This process is known as **ETL (Extract, Transform, Load)**. In Power BI, the engine responsible for this is **Power Query**. Mastering Power Query ensures that your data models are efficient, accurate, and ready for analysis.

---

## 1. Data Extraction (Get Data)
Power BI can connect to hundreds of different data sources. The extraction phase involves pulling raw data into the Power Query environment.

* **Flat Files:** CSV, Excel (`.xlsx`), Text files, JSON, XML.
* **Databases:** SQL Server, MySQL, PostgreSQL, Oracle, Amazon Redshift.
* **Cloud & Web:** SharePoint, Google Analytics, Salesforce, Web Pages (scraping HTML tables).

**Best Practice:** Always connect to the most upstream, reliable data source available (e.g., connect directly to a SQL database rather than exporting to Excel and then connecting to the Excel file).

---

## 2. The Power Query Editor Interface
Once you click "Transform Data" after connecting to a source, you enter the Power Query Editor. Key areas include:
* **Queries Pane (Left):** Lists all the different data sources and tables you have connected to.
* **Data View (Center):** A preview of your data based on the current step.
* **Query Settings & Applied Steps (Right):** The most critical pane. Power Query records every transformation you make as a chronological "step". You can undo, modify, or reorder these steps without affecting the original source data.

---

## 3. Basic Data Transformations (Cleaning)
These are the foundational tasks required to clean messy datasets.

* **Promote Headers:** Often, the first row of raw data contains the column names. Power Query allows you to push the first row up to become the official column headers.
* **Changing Data Types:** Ensuring a column is recognized correctly (e.g., changing a date stored as "Text" into "Date", or changing a financial figure into "Fixed Decimal Number"). This is crucial for DAX calculations later.
* **Removing/Keeping Rows and Columns:** * Strip out null rows, error rows, or unnecessary metadata rows at the top of a file.
    * Remove columns that aren't needed for analysis to reduce the file size and improve performance.
* **Replacing Values & Splitting Columns:**
    * Replace specific errors or text strings (e.g., replacing "N/A" with `0`).
    * Split a single "Full Name" column into "First Name" and "Last Name" using a delimiter like a space or comma.

---

## 4. Advanced Transformations (Shaping)
Shaping involves altering the fundamental structure of your tables to prepare them for relational data modeling (Star Schema).

### Merging Queries (Like SQL JOINs)
Merging adds *columns* to a query by linking two tables based on a common key (e.g., matching a `CustomerID` in a Sales table to a `CustomerID` in a Customers table).
* Power Query supports Left Outer, Right Outer, Inner, Full Outer, Left Anti, and Right Anti joins.

### Appending Queries (Like SQL UNION)
Appending adds *rows* to a query. For example, if you have 12 separate Excel files for each month of sales, you can Append them into one massive "Annual Sales" table.

### Unpivoting Data
Many raw reports (like Excel financial statements) are wide (e.g., a row for a product, and separate columns for Jan, Feb, Mar, Apr...). Data models prefer "long" data. **Unpivoting** converts those month columns into two standardized columns: "Attribute" (the month) and "Value" (the sales amount).

### Conditional Columns
You can create new columns based on `IF/THEN` logic directly in Power Query without writing complex code.

```powerquery
// Example Logic built via the UI:
If [SalesAmount] > 10000 Then "High Value"
Else If [SalesAmount] > 5000 Then "Medium Value"
Else "Low Value"
```

## 5. Introduction to M Code (The Power Query Formula Language)
Behind the graphical interface of Power Query, every click generates code. This language is called **M**. While you can do 95% of your work using the UI buttons, understanding M allows for advanced, dynamic transformations.

You can view the M code by clicking **Advanced Editor** in the Home tab.

### M Code Syntax Structure
M code operates using a `let` block (where variables and steps are defined) and an `in` block (which outputs the final result).

```powerquery
let
    // Step 1: Connect to the source
    Source = Csv.Document(File.Contents("C:\Data\Sales.csv"),[Delimiter=",", Columns=5, Encoding=1252, QuoteStyle=QuoteStyle.None]),
    
    // Step 2: Promote Headers
    #"Promoted Headers" = Table.PromoteHeaders(Source, [PromoteAllScalars=true]),
    
    // Step 3: Change Data Types
    #"Changed Type" = Table.TransformColumnTypes(#"Promoted Headers",{{"OrderID", Int64.Type}, {"Date", type date}, {"Revenue", Currency.Type}})
in
    #"Changed Type"
```

**Key Takeaways regarding M:**
* It is **case-sensitive** (unlike DAX or SQL).
* Each step refers to the name of the *previous* step (e.g., the `#"Changed Type"` step transforms the data generated by the `#"Promoted Headers"` step).
