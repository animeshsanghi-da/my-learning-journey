# Power BI Data Modeling & Relationships (Star Schema)

## Overview
Data Modeling is the "engine room" of Power BI. It involves connecting multiple data sources, defining how they relate to one another, and structuring them to support efficient analysis. A well-designed model ensures that your reports are accurate, performant, and easy to maintain.

---

## 1. The Star Schema Concept
The industry standard for Power BI data modeling is the **Star Schema**. It organizes data into two types of tables:

### Fact Tables
* **Content:** These contain quantitative data (metrics) such as Sales Amount, Quantity, or Temperature.
* **Characteristics:** They are usually very long (millions of rows) and contain foreign keys that link to Dimension tables.
* **Example:** `FactSales` containing `OrderDateID`, `ProductID`, `CustomerID`, and `SalesAmount`.

### Dimension Tables (Lookup Tables)
* **Content:** These contain descriptive attributes about the facts, such as Product Names, Customer Cities, or Calendar dates.
* **Characteristics:** They are usually wider and shorter than fact tables.
* **Example:** `DimProduct` containing `ProductID`, `ProductName`, `Category`, and `Sub-Category`.



---

## 2. Managing Relationships
Relationships define how data flows between tables. Power BI uses these links to filter and calculate data across different dimensions.

### Cardinality
* **One-to-Many (1:*):** The most common and recommended type. One record in a Dimension table (e.g., one Product ID) matches many records in a Fact table (e.g., many sales of that product).
* **One-to-One (1:1):** Used rarely, often when a table is split for security or performance reasons.
* **Many-to-Many (*:*):** Should be avoided where possible as it introduces ambiguity; often resolved using a "Bridge Table."

### Cross-Filter Direction
* **Single Direction:** The filter flows from the Dimension table to the Fact table. This is the safest and most efficient setting.
* **Both Directions (Bi-directional):** Allows filtering in both ways. While powerful, it can lead to performance issues and "ambiguous" relationships where the engine doesn't know which path to take to filter data.

## 3. The Power of the Date Table (Calendar Table)
A dedicated Date Table is essential for **Time Intelligence** (comparing this year vs. last year, YTD, etc.).
* It must contain one row for every day of the year.
* It must have no gaps in dates.
* It should contain columns like `Year`, `Month`, `Quarter`, `Week Number`, and `Day of Week`.

### Creating a Date Table with DAX
```dax
DateTable = 
ADDCOLUMNS (
    CALENDARAUTO(),
    "Year", YEAR([Date]),
    "Month Name", FORMAT([Date], "MMMM"),
    "Month Number", MONTH([Date]),
    "Quarter", "Q" & FORMAT([Date], "Q"),
    "Weekday", FORMAT([Date], "dddd")
)
```

## 4. Calculated Columns vs. Measures
Understanding where to put your logic is critical for model performance.

| Feature | Calculated Column | Measure |
| :--- | :--- | :--- |
| **When it Calculates** | During Data Refresh | During Visual Interaction (on-the-fly) |
| **Storage** | Stored in the model (increases file size) | Not stored (calculated in RAM) |
| **Context** | Row Context (looks at current row) | Filter Context (looks at the visual's filters) |
| **Best Use** | Slicers, Row-level attributes | Aggregations (Sum, Avg), KPIs |

---

## 5. Performance Optimization Tips
* **Remove Unnecessary Columns:** Only import what you need. High-cardinality columns (like unique IDs or timestamps) significantly increase model size.
* **Prefer Measures over Columns:** Use measures for calculations to keep the file size small.
* **Use Integer Keys:** Relationships perform faster when linking integer columns (ID numbers) rather than text columns (Names).
* **Hide Foreign Keys:** Hide the "ID" columns in your Fact table from the "Report View" to prevent users from using them in visuals instead of the descriptive names in Dimension tables.

3. [ ] Have I marked my Date Table as a "Date Table"?
4. [ ] Are all my measures organized into a dedicated "Measures Table"?
